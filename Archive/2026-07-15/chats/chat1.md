**Company description:** # PLNLY Creative Brief — Visual Identity (for Claude Design)

## The brand in one breath

**PLNLY** (written, stylized wordmark), spoken **"Plainly."** A personal-AI setup, teaching, and retainer service — Brooklyn/NYC first, then online. It sets the tools of modern life up around how a household actually lives, teaches the household to run them, and hands over the keys.

- **Signature line (locked):** Built bespoke. Yours to run.
- **Essence line:** a beautiful plainness.
- **First-contact rule:** every new surface pairs **PLNLY** with **Plainly** at first sight, so the stylization never stands alone unexplained. "Plainly" sits beneath the wordmark as a designed pronunciation reveal — not a fallback.

## Who it's for

Two segments, Brooklyn/NYC first, then online.

1. **Affluent, time-constrained households.** Busy parents, frequent travelers, families who could figure the tools out themselves and would rather not lose the weekend to it. They want AI configured around how they live — scheduling, groceries, bills, travel, household ops — while staying in control of their own systems. Buying back time and capability, not hiring a butler.
2. **Parents of college-bound teens.** Want their kids fluent and responsible with AI tools before campus — NotebookLM, citations, academic-integrity lines. Tool literacy, scoped tight. Not subject tutoring.

The service reads as a capability, not a four-item menu. A household should see its own need in it — travel is the clearest example: never the headline, always at home inside the service.

## Positioning — the one thing that makes it ours

Every competitor delegates: they run the AI and keep the keys. PLNLY hands the keys over — sets it up, teaches the household to run it, leaves them owning the capability rather than dependent on us. Ownership, not concierge. This is the moat, and it should be felt in the identity, not only stated in copy.

## Personality

Capable first, then warm, with edges — all of it governed by restraint.

**One line:** the capable friend with great taste — treats you as a peer, says it once and says it well, never raises its voice.

- **Capable, not reassuring.** Assumes you're smart and busy, not worried. Never explains down.
- **Warm, with edges.** The warmth of being treated as an equal, not a hug. A dry, occasional, never-mean edge keeps it off both customer-service cheer and concierge gloss.
- **Restraint is the volume knob.** Capability shown not announced; warmth in one well-chosen word; the edge as a soft aside. Greige-grounded calm — a beautiful plainness, never sparse or clinical.

**Executable contrasts:**

- Says "Here's the setup. You've got it from here." Never "Don't worry — it's so easy!"
- One dry aside, rarely. Never a brand-voice "joke" reaching to be relatable.
- One restrained accent on a greige ground. Never a palette doing the talking.
- Names the real tool and the real task. Never "AI-powered solutions."

## Voice (for copy on any surface)

Plain language with one elevated word doing real work. Specific over generic — name the tool, the task, the place. Short where it can be; rhythm where it earns it. No hype, no wellness/LinkedIn buzzwords, no startup-cheer. Confidence by what's left out.

## Visual direction

**The ground is greige.** Not white, not cold; not warm-and-fluffy or yellow. Greige leads, with warm neutrals and soft blues, greens, and greys layered in, and one confident, sparing accent (coral, orange, warm yellow, or muted pink — one, used rarely). Calm, not empty. Sleek and simple — a beautiful plainness.

**Type.** Design-forward and quiet. Not generic SaaS sans-serif-plus-gradient; not twee or hand-lettered. The warmth should read as competence, not as dumbing-down.

**Logo.** The PLNLY wordmark with "Plainly" set beneath as a pronunciation reveal — the core lockup. Build it as a system, not one static mark: it has to flex across surfaces and, eventually, sub-brands.

**References (feel, not copying):** Aesop packaging (restraint, uniformity, quiet mystery); Park Hyatt / Edition (warm, sleek, considered); Aer, Bellroy, Shinola (quality-everyday-object, no logo-screaming).

**Hard avoids:**

- Tech-bro SaaS blue-and-white
- Gradient mesh backgrounds
- AI clichés — circuit patterns, neural-network line art, robots
- Startup-generic / corporate-cold
- White airy-minimalism — the brand is calm, not sparse

## Brand architecture — design for the family, not the only child

PLNLY is the umbrella. Service lines will become sub-brands beneath it. The logo, color, and type system must have enough flex that a sub-brand shares clear DNA with PLNLY without being identical — the way a hotel group's properties carry their own names and feel while plainly belonging to the same family. Design the umbrella now so it can hold the others later. Only the umbrella is live today.

Design systems are folders on the file system containing typography guidelines, colors, assets, brand style and tone guides, css styles, and React recreations of UIs, decks, etc. They give design agents the ability to create designs against a company's existing products, and create assets using that company's brand. Design systems should contain real visual assets (logos, brand illustrations, etc), low-level visual foundations (e.g. typography specifics; color system, shadow, border, spacing systems), reusable UI components, and high-level UI kits (full screens).

No need to invoke the create_design_system skill; this is it.

An automated compiler reads this project, bundles the components into a runtime library, and indexes the styles. It discovers everything from file content and sibling relationships — not from folder names — so the only fixed location is:

- `styles.css` at the project root (or `index.css` / `globals.css` / `global.css` / `main.css` / `theme.css` / `tokens.css` — first match wins). This is the global-CSS entry point; consumers link this one file. Keep it as a list of `@import` lines only. Everything it transitively `@import`s is shipped to consumers; `@font-face` rules anywhere in that closure declare the webfonts.

Organize everything else however suits the brand. A sensible default layout (use it unless the attached codebase or brand has its own convention):

- `tokens/` — CSS custom properties, one file per concern (`colors.css`, `typography.css`, `spacing.css`, …), each `@import`ed from `styles.css`.
- `components/&lt;group&gt;/` — reusable React UI primitives.
- `ui_kits/&lt;product&gt;/` — full-screen click-through recreations of real product views.
- `guidelines/` — foundation specimen cards and deeper-dive prose.
- `assets/` — logos, icons, illustrations, imagery.
- `readme.md` (root) — the design guide and manifest.

What the compiler looks for, regardless of path:
- A **component** is any `&lt;Name&gt;.jsx` / `&lt;Name&gt;.tsx` (PascalCase stem) with a sibling `&lt;Name&gt;.d.ts` in the same directory. Add `&lt;Name&gt;.prompt.md` alongside, and one `@dsCard`-tagged `.html` per directory (its first line is `&lt;!-- @dsCard group="…" --&gt;`; details under "Components" below).
- A **token** is any `--*` custom property declared under `:root` (or a single-selector theme scope) in a file reachable from `styles.css`.
- A **font** is any `@font-face` rule in that same closure; its `src: url(…)` targets are the binaries shipped to consumers.

Components
- These are the brand's reusable UI primitives — Button, IconButton, Input, Select, Checkbox, Radio, Switch, Card, Badge, Tag, Avatar, Tabs, Dialog, Toast, Tooltip, etc. Group by concern (e.g. `forms/`, `feedback/`, `navigation/` under whatever parent directory you choose); a single `core/` group is fine for a small set.
- Each component is one file `&lt;Name&gt;.jsx` (or `.tsx`) with `export function &lt;Name&gt;(props) {…}` — a named, PascalCase export; that name becomes the public API and the literal `export` keyword is required so the bundler picks it up. Keep them self-contained: import React only, reference styling via the CSS custom properties (no CSS-in-JS libs, no npm packages). Siblings may import each other with relative paths.
- In the same directory, write `&lt;Name&gt;.d.ts` with the props interface — the sibling `.d.ts` is what gives a component its props contract, adherence rules, and starting-point eligibility; a `.jsx` without one is still bundled and exported under the namespace but gets none of those — and `&lt;Name&gt;.prompt.md` (first line is a one-sentence "what &amp; when", then a small JSX usage example, then notable variants/props).
